# Internacionalização (i18n)

## Idiomas suportados

| Locale | Status MVP |
|---|---|
| `pt-BR` | ✅ default |
| `en-US` | 🟡 v2 (estrutura pronta, traduções pendentes) |
| `es-ES` | 🟡 v2 |

## Stack

- **next-intl** (App Router) — message format ICU, type-safe.
- Mensagens em `apps/<app>/messages/<locale>.json`.
- Mensagens compartilhadas em `packages/messages/<locale>.json`.

## Estrutura

```json
// messages/pt-BR.json
{
  "common": {
    "save": "Salvar",
    "cancel": "Cancelar",
    "back": "Voltar"
  },
  "documents": {
    "newInternal": "Cadastrar documento interno",
    "form": {
      "title": "Título",
      "category": "Categoria"
    }
  }
}
```

## Detecção de locale

1. **Cookie do user** — preferência salva (`locale=pt-BR`).
2. **Header `Accept-Language`** — fallback no primeiro acesso.
3. **`pt-BR`** — default global.

## Fora de mensagens (formatação)

- **Datas**: `Intl.DateTimeFormat` com locale do user. Default `dd/MM/yyyy` em pt-BR.
- **Números**: `Intl.NumberFormat`.
- **Pluralização**: ICU `{count, plural, one {1 item} other {# itens}}`.

## Conteúdo do tenant não é traduzido

Categorias de documento, nomes de processo, descrições de NC etc. são entradas livres do tenant. **Não traduzimos**. UI mostra como cadastrado.

## Email templates

- Templates em `packages/notifications/templates/<template>/<locale>.tsx`.
- Worker escolhe o template baseado no `user.locale`.
- Fallback `pt-BR` se locale não tiver template.

## Audit log

- `action`, `entity_type`, etc. — sempre em inglês (chaves técnicas).
- Tela mostra tradução com `t('audit.action.docs.published')`.

## Testes

- E2E roda em `pt-BR` por default.
- Smoke test em outros locales para garantir que nenhuma chave fica faltando.
- ESLint regra: nenhum literal de UI fora de `t()` (custom rule).

## Adicionar novo idioma

1. Criar `messages/<locale>.json` em todos os apps + `packages/messages`.
2. Traduzir todas as chaves (ferramenta sugerida: Lokalise ou Crowdin).
3. Adicionar à lista no admin de cadastro de usuário.
4. Testar smoke E2E.
